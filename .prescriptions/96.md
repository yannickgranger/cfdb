## Decisions

### 1. `short_rev` keyspace-name derivation — REUSE

- **REUSE** `commands::short_rev(rev: &str) -> String` at `crates/cfdb-cli/src/commands.rs:117`
- **Why reuse:** identical decision. Any SHA that parses from `<url>@<sha>` is still a 40-char hex string that needs truncation to 12. If the hex-truncation rule changes (e.g., bump to 14 chars), both paths must change together — by definition the same decision.
- **How to wire:** `extract_at_url_rev` calls `short_rev(sha)` when `--keyspace` is not provided (mirrors `extract_at_rev` at `commands.rs:90`).
- **Not forbidden move #2** (forking `short_rev`) because: we route through the single existing function, not copy its body.
- **Forbidden-move intersection:** `#2 Do not fork short_rev` — this decision EXACTLY satisfies that rule.

### 2. `GitWorktree` RAII guard — NO REUSE, NO FORK (does not apply to URL path)

- **DECISION: NEITHER REUSE NOR FORK.** `GitWorktree` is tied to the `git worktree add --detach` semantic — a feature of an existing local repository. The URL@SHA path does NOT have a pre-existing local repo; it performs a fresh `git clone --depth 1` (or equivalent) into a cache directory. There is no `git worktree` step in the URL path. `GitWorktree` stays the canonical manager for the same-repo flow (forbidden move #1 untouched).
- **Why this is NOT forking `GitWorktree`:** we are not creating a parallel worktree manager. We are creating a cache-backed clone-once directory, which is a different operation with different lifecycle (persistent, not RAII-cleaned).
- **Why this is NOT a forbidden move #1 violation:** `GitWorktree` remains the sole caller of `git worktree add`; the URL path calls `git clone`, which is a different subprocess verb with different semantics (new repo initialization vs. in-place worktree attachment).
- **Reviewer hint:** If a future refactor unifies the two paths, it MUST do so in `GitWorktree` (extend) — not by creating a parallel `GitClone` guard. That is a MOVE/REFACTOR out of scope for this issue.

### 3. `extract` dispatcher (`--rev` branch) — EXTEND

- **EXTEND** `commands::extract` at `crates/cfdb-cli/src/commands.rs:26-37` by adding a URL-vs-SHA discrimination inside the `Some(rev)` branch.
- **What's missing:** today the `Some(rev)` arm unconditionally calls `extract_at_rev` (requires local repo). Option W adds a second branch: `if rev.contains('@') && looks_like_url(rev) → extract_at_url_rev`.
- **Why extend, not create a parallel dispatcher:** same bounded context (CLI command dispatch), same clap `Option<String>` input, same termination point (`extract_at_path` on a local directory). A parallel dispatcher would split the `--rev` flag's semantics across two call sites — exactly the "layer addition" anti-pattern in §5h. The dispatcher is the canonical discriminator.
- **Signature after extend** (unchanged externally):
  ```rust
  pub fn extract(
      workspace: PathBuf,
      db: PathBuf,
      keyspace: Option<String>,
      hir: bool,
      rev: Option<String>,
  ) -> Result<(), crate::CfdbCliError> {
      match rev.as_deref() {
          None => extract_at_path(&workspace, &db, keyspace, hir),
          Some(rev) if is_url_at_sha(rev) => extract_at_url_rev(rev, &db, keyspace, hir),
          Some(rev) => extract_at_rev(&workspace, rev, &db, keyspace, hir),
      }
  }
  ```
- **Wiring assertion:** the `Some(rev)` branch MUST be a match-with-guard, not two sequential `if` statements. See §WIRING ASSERTION below.
- **Forbidden-move intersection:** none — `extract_at_rev` is NOT modified, the subprocess contract for same-repo flow stays verbatim (forbidden move #4 untouched).

### 5. `extract_at_url_rev` — CREATE

- **CREATE** `fn extract_at_url_rev(url_at_sha: &str, db: &Path, keyspace: Option<String>, hir: bool) -> Result<(), CfdbCliError>` in `crates/cfdb-cli/src/commands.rs`
- **BOUNDED_CONTEXT:** `cfdb-cli::commands` — private function sibling to `extract_at_path` and `extract_at_rev`.
- **Why create:** the URL@SHA path is a different decision from the same-repo path.
  - Same-repo: pre-existing local repository → `git worktree add --detach` + `extract_at_path` + cleanup. Cache is the local `.git` — no external state.
  - URL@SHA: no local repo → `git clone` into `~/.cache/cfdb/extract/<hash(url)>/<sha>/` + `extract_at_path` against that tree. Cache IS the whole point (AC-3).
  - If the same-repo worktree strategy changes (e.g., switch to `git archive | tar`), the URL path should NOT change — it already materialized a full tree via clone. Independent decisions.
- **Not duplication because:** `extract_at_rev` and `extract_at_url_rev` share nothing but the call to `extract_at_path(worktree_path, db, ks, hir)` at the end. That shared call is the canonical extractor entrypoint — reuse, not copy.
- **Name collision check (Step 5d):**
  ```bash
  rg "fn\s+extract_at_url_rev" crates/*/src/ -l
  # → zero hits (verified 2026-04-21)
  ```
- **Shape prescription (structure, not code body):**
  1. Split `url_at_sha` via `parse_url_at_sha` → `(url, sha)`. Return `CfdbCliError::Usage` on `None`.
  2. Compute cache dir via `cache_dir_for(url, sha)` (decision 6).
  3. If cache dir exists and is a readable git tree (presence sentinel file, e.g. `.cfdb-extract-ok` written after successful clone, gate off that — do NOT rely on git-internal state): skip clone.
  4. Otherwise: create parent dir, run `git clone --depth 1 --quiet <url> <cache_dir>` then `git -C <cache_dir> fetch --depth 1 origin <sha>` then `git -C <cache_dir> checkout --quiet <sha>`. Write sentinel file on success.
     - **Why `--depth 1`:** network efficiency; we only need the one SHA's tree.
     - **Why `fetch --depth 1 origin <sha>`:** `git clone <url>` does not fetch a non-default SHA. Depending on server config, `git fetch --depth 1 origin <sha>` needs `uploadpack.allowReachableSHA1InWant=true` on the server — Gitea has it on by default. If the fetch fails with a stderr hint about `allowReachableSHA1InWant`, the CLI error message MUST name this as a server-config issue (not a user error).
  5. `ks_name = keyspace.unwrap_or_else(|| short_rev(sha))`.
  6. Call `extract_at_path(&cache_dir, db, Some(ks_name), hir)`.
- **Credentials (AC-2):** subprocess `git clone`/`fetch` inherit ambient git credentials — SSH agent keys (`$SSH_AUTH_SOCK`), `~/.config/git/credentials`, `GIT_ASKPASS`, and Gitea tokens configured via `git config --global credential.helper`. No new plumbing. This discharges AC-2 without adding a deps.
- **Error handling:** subprocess failures (clone, fetch, checkout) return `CfdbCliError::Usage` with a message containing the subprocess's stderr, so the user sees the real git error (e.g., "Authentication failed for 'https://...'" or "reference not found").
- **Forbidden-move intersection:**
  - #4 (subprocess contract same-repo) — not violated; we use subprocess `git clone` here, and `extract_at_rev` is unchanged.
  - #3 (no unconditional `git2`) — not violated; we use subprocess `git`, not `git2`. `git2` remains gated behind `git-enrich`.
  - #6 (no cache in `target/`/`tempdir`) — not violated; we use a persistent cache dir (decision 6).

## Council-specific notes

- **Subprocess contract preserved** — same-repo `extract_at_rev` unchanged. URL path adds `git clone`/`fetch`/`checkout` subprocess calls, which is a continuation of the subprocess-as-contract design, not a break.
- **Ambient-credentials contract** — AC-2 is satisfied without new dependency or config. Users configure git credentials per their normal workflow (`git config credential.helper`, SSH keys, Gitea tokens set once). This is the same UX as `git clone` itself.
- **`sha2` as a new workspace dep** — small footprint, pure-Rust, commonly already present in transitive trees. Acceptable addition for hashing URLs → cache keys. Alternative (`twox-hash`, `xxhash-rust`) is faster but less auditable; `sha2` is the conservative pick given the tool's anti-drift stance.
- **Schema version** — no change. Extracted facts are identical; only the input-source path changes.
- **Architect-lens routing for the RFC (if filed retroactively):** this issue was filed downstream of qbot-core council-4046 without a dedicated cfdb RFC. Per CLAUDE.md §1, new-capability work is "RFC-first". Recommend: add a short §A1.7 amendment to `docs/RFC-032-v02-extractor.md` documenting "URL@SHA variant of --rev" as the prescribed addition, filed in the same PR as the implementation. This discharges the RFC-first invariant without a full new RFC (the design is a natural extension of the already-ratified §A1.6 `--rev` flag).
