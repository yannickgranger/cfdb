## Forbidden moves (from .context/236.md)

1. **Do not mutate `ClassifyEnvelope` struct in `cfdb-query`.** Rationale: AC6 regression lock + #213 compositional-envelope lesson ("ClassifyEnvelope is a shipped wire contract"). The new format is a serialization choice, not a new wire field.
2. **Do not change `--format json` default output bytes.** Rationale: AC4 regression lock; existing integration tests (`classify_self_dogfood.rs`) assert on JSON shape. Adding a new format must not perturb the JSON branch.
3. **Do not use `HashMap`, `sort_unstable`, or any wall-clock-dependent serialization.** Rationale: RFC §12.1 G1 byte-determinism contract; `canonical_dump.rs:16-44` documents the invariants. Classify's sorted-jsonl output shares the same two-run byte-identical guarantee.
4. **Do not embed routing hints (`fix_skill` or equivalent) in finding rows.** Rationale: carried over from #213 — RFC-cfdb.md §A2.3 + line 919 council BLOCK-1 (solid-architect), enforced by `finding_no_skill_field` architecture test. Routing lives in `.cfdb/skill-routing.toml`. Serializing findings to sorted-jsonl must not leak routing columns.
5. **Do not add a new `pub struct` or `pub enum` in `cfdb-query` without a corresponding `specs/concepts/cfdb-query.md` entry.** Rationale: user-memory `feedback_graph_specs_pub_types.md` — `make graph-specs-check` blocks `/ship` on undeclared pub types. **This rule is trivially satisfied if AC6 holds (no struct changes).**
6. **Do not invoke `/ship` with `--gate-profile`.** Rationale: session-handoff lesson — the installed `quality-ship` binary rejects the flag; skill template is stale.

Each emitted object carries a `kind: "node" | "edge"` discriminator so a stream-parser does not need positional context. (The original RFC draft named this verb `cfdb export --format=sorted-jsonl`; the shipped CLI uses `cfdb dump` — a one-name verb is cleaner than a verb+flag pair with a single legal flag value, and the format itself is unaffected. **Ambiguity 1 — AC3 sort key mentions `rule`; no `rule` field on `Finding`.**

Issue AC3: *"Lines sorted by `(class, qname, rule)` lexicographically — deterministic, byte-stable across runs"*

Facts:
- `Finding` struct at `crates/cfdb-query/src/inventory.rs:109-119` has fields `{qname, name, kind, crate_name, file, line, bounded_context}` — no `rule`.
- `Finding` derives `Ord` at line 109 — sort order is declaration order: `(qname, name, kind, crate_name, file, line, bounded_context)`.
- `findings_by_class: BTreeMap<DebtClass, Vec<Finding>>` — iteration is `(class, [findings-in-vec-order])`. Vecs are populated by `run_classifier_rule` at `scope/classifier.rs:46` — order reflects cypher query result order, NOT derived-Ord order, unless the producer explicitly sorts.
- The `classifier_rules()` list at `scope/classifier.rs:18-39` provides a 1:1 map `DebtClass → cypher source`. There is no separately-identifiable "rule" beyond the class.

Interpretations (for `/prescribe` to resolve — NOT a prescription):
- "rule" in AC3 may be **synonymous with "class"** (each class → one cypher rule 1:1). Sort key degenerates to `(class, qname)` plus `Finding::Ord` tie-breakers.
- OR "rule" may be a **forward-compatible placeholder** anticipating a future world where one class has multiple rules (cfdb-recall corpus extension). In that case the `Finding` struct would need a `rule` field — but that's forbidden-move #1 (mutate ClassifyEnvelope / its Finding members).

Prescription call: (a) clarify the AC in the issue body as "sort by `(class, qname)` plus the derived `Finding::Ord` tail", OR (b) change AC3 language to match what `Finding`'s derived `Ord` already provides.

**Ambiguity 2 — AC5 metadata shape: `#`-prefix header line vs sidecar JSON.**

Issue AC5 explicitly defers the choice to `/prescribe`: *"Envelope-level metadata (`schema_version`, context, diff_source) emitted consistently with `cfdb dump` conventions"*.

Facts:
- `cfdb dump` (`aux.rs:12-17`) emits ONLY node+edge lines, no envelope header. "Consistent with cfdb dump" would mean no header — but then `schema_version` and `diff_source.{a,b,restrict_count}` have no home in the sorted-jsonl output.
- `cfdb diff --format sorted-jsonl` (`diff.rs:85-110`) emits a first line `{"op":"header","a":...,"b":...,"schema_version":...}` followed by per-fact lines with `op: added|removed|changed|warning`. This is the **shipped cfdb precedent for an envelope with scalar metadata on a sorted-jsonl channel** — the `#` comment-prefix alternative the issue body mentions is NOT used anywhere in cfdb today.
- `cfdb diff`'s header carries only `(a, b, schema_version)`. Classify's envelope carries `(schema_version, inventory.context, inventory.keyspace_sha, diff_source.{a, b, restrict_count}, warnings)` — richer scalar surface. `warnings` is a `Vec` not a scalar.

Prescription call: either (a) emit a single `{"op":"header",...}` first line with scalar metadata, per-Finding lines with `{"op":"finding", "class":..., "qname":..., ...other Finding fields}`, and per-warning lines with `{"op":"warning", "kind":..., "message":..., "suggestion":...}` — closely mirroring diff; OR (b) the `#` comment shape (not used elsewhere in cfdb). Prescription MUST pick one consistent with an existing cfdb emit to avoid a third pattern.
