## Forbidden moves (from .context/127.md)

1. **Do NOT place the crate inside any existing `crates/cfdb-*/`** — it is `tools/check-prelude-triggers/`, a standalone workspace member. Rationale: clean-arch boundary test in cfdb (`cfdb-hir-extractor/tests/arch_boundary.rs` pattern) — tool is NOT a cfdb component.
2. **Do NOT use `cargo_metadata` crate** — parse `Cargo.toml` directly with `toml`. Rationale: RFC-034 rust-systems Amendment 2; `cargo_metadata` invokes `cargo` as subprocess which re-enters workspace lock resolution and deadlocks in nested CI contexts.
3. **Do NOT add per-trigger boolean fields to JSON** — use `triggers_fired: []` array only. Rationale: RFC-034 SOLID Amendment A1 (OCP); future Tier-2 promotions append IDs without breaking consumers.
4. **Do NOT convention-bake consumer-repo paths** — every TOML path is an explicit CLI argument. Rationale: RFC-034 clean-arch Amendment A; binary must be consumable by any repo that follows the S0 schema.
5. **Do NOT let the binary write files or maintain implicit caches** — stateless. Rationale: RFC-034 §4.2 ("binary is entirely stateless; cached output lives under consumer-repo `_generated/`, not in cfdb").
6. **Do NOT skip the cfdb `CLAUDE.md` §2.5 four-row Tests block in the PR body** — Unit + Target-dogfood required with real content; Self-dogfood + Cross-dogfood must either carry real content OR `none — rationale: <why>`. Rationale: cfdb PR merge gate.
7. **Do NOT confuse RFC-034 C-triggers with cfdb Cypher Triggers T1/T3** — they are distinct concepts. C-triggers are mechanical pre-council gates; T1/T3 are graph query patterns. This binary implements C-triggers only.
8. **Do NOT introduce dependencies on `cfdb-core`, `cfdb-extractor`, or any other cfdb crate** — the tool is standalone. Rationale: issue AC-5 + clean-arch architect gate.
