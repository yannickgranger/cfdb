  Searches exhausted: `rg 'is_cross_context|cross_context' crates/` → no hits; `rg 'n_contexts' crates/` → no hits.

- Adding a sibling `t3-concept-multi-crate.cypher` keeps `hsb-by-name.cypher` frozen for the existing `scope.rs` consumer → zero regression risk for the scope verb.
- Extending `hsb-by-name.cypher` in place creates additive columns that the scope consumer ignores → zero behavior change for scope, but the shape change is semantically load-bearing (hsb-by-name.cypher was called out in RFC §13 as the Pattern A v0.1 gate).
- Prescription decides. Both are consistent with the Forbidden moves list.

### Ambiguity A2 — `canonical_candidate` computation

**Observation (fact):**
- Issue AC defines `canonical_candidate` := item in crate matching the concept's `canonical_crate`. This requires a per-name JOIN against `:Context.canonical_crate`.
- The existing scope.rs `query_canonical_candidates` uses a DIFFERENT semantic — it filters hsb-by-name rows to items whose `crate` is in the context — producing `CanonicalCandidate { name, kind, crates, qnames, files }` records, NOT a single `canonical_candidate: String` column.
- The issue's canonical_candidate column says "the item in the crate matching the concept's canonical_crate if any" — for a name appearing in crates [A, B, C], find the crate K such that there exists some `:Context c` with `c.canonical_crate = K`, then the canonical_candidate is K (or the item's qname within K).
- Live graph probe: `MATCH (c:Context) RETURN c.canonical_crate` on cfdb returns `"cfdb-core"`. So a T3 row for a name present in [cfdb-cli, cfdb-core, cfdb-extractor] → canonical_candidate="cfdb-core" (because cfdb-core is the `:Context.canonical_crate`).
- Cypher-level JOIN: `MATCH (a:Item) WITH a.crate AS crate, a.name AS name OPTIONAL MATCH (c:Context) WHERE c.canonical_crate = crate RETURN crate, c.canonical_crate` — runs in one pass. But see OPTIONAL MATCH WHERE limitation documented in #101: this pattern drops rows where no match found. Different from T1 because T3 just needs to pick ONE canonical candidate per name (if any) — Rust can iterate the crates and look up canonical set.

**Divergence verdict (fact, not prescription):**
- Computing `canonical_candidate` purely in cypher is fragile under the v0.1 parser's OPTIONAL MATCH limitation.
- The same ergonomics demonstrated in #101 (cypher dumps the raw inventory, Rust applies per-row correlations) applies cleanly here: fetch the set of canonical_crates once (`MATCH (c:Context) RETURN c.canonical_crate`), then for each T3 row mark `canonical_candidate` = the first entry of `crates[]` that appears in the canonical set (or null).
- Prescription decides whether canonical_candidate lives at cypher layer or Rust layer. Either way produces the same output shape.

### Ambiguity A3 — T3 kind filter (struct/enum/trait vs include fn/method)

**Observation (fact):**
- `hsb-by-name.cypher` restricts to `struct/enum/trait` — rationale: *"Fn items are excluded because method-name collisions across unrelated types are semantic noise."*
- `classifier-context-homonym.cypher` explicitly handles fn/method via `signature_divergent` UDF.
- Issue #102 body does NOT specify a kind filter — it says *"for each distinct `:Item.name` that appears in items belonging to ≥2 distinct `:Crate`s"*. Without a kind filter, fn/method collisions would flood T3 output.
- Doctrinal default (hsb-by-name header line 42-46): struct/enum/trait is the Pattern A load-bearing scope.

**Divergence verdict (fact, not prescription):**
- The issue AC implicitly inherits hsb-by-name's kind restriction since T3 is framed as "the underlying Pattern A query exposed as an explicit Trigger with CLI surface". Inheriting the restriction preserves compatibility with the doctrinal Pattern A semantics.
- If prescription chooses to widen to fn/method, doing so diverges from the classifier's split-responsibility model: today `classifier-context-homonym` handles fn/method via signature divergence, and `classifier-duplicated-feature` handles struct/enum/trait via same-context filter. Mixing fn into Pattern A raw would duplicate the context-homonym signal.
