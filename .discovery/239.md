**`syn::Type` ‚Üí rendered String ‚Üí resolved qname** (single linear flow per queue entry; no forks, no diverges today)
  ENTRY:     `syn::Type` (from syn AST, owned by the per-file visitor)

**Where `render_type_inner` plugs in (prescriber's choice):**

  - Option B (resolve time): extend resolver to textually post-process the rendered string (parse back `"Vec"` / `"Option"` etc. against stored inner candidates).

**No [FORK] or [DIVERGE] markers.** The existing pipeline is a single linear flow; the new unwrapping is an extension, not a competing parallel path.

## AMBIGUITIES ‚Äî Prescribe must resolve

**Ambiguity A ‚Äî Where does unwrapping happen? (queue time vs resolve time)**
  - **A1 ‚Äî Queue time.** Each `deferred_returns.push` / `deferred_type_of.push` call extends to produce both the outer render and the unwrapped inner(s). Requires changing queue tuple shape: `Vec<(String, String)>` ‚Üí `Vec<(String, Vec<String>)>` or `Vec<(String, String, Vec<String>)>`. Touches 4 push sites + the resolver + the `Emitter` struct definition.
  - **A2 ‚Äî Resolve time (textual).** Resolver post-processes the existing rendered string: if string starts with `Vec(` / `Option(` / `Result(` / etc., unwrap textually. Requires render-format awareness (what does `render_type_string` produce for `Vec<Foo>`? Today it produces `"Vec"` ‚Äî there's no generic-arg render left to parse at resolve time). **A2 is likely infeasible** unless `render_type_string` is ALSO extended to emit a different format when an unwrapper is available.
  - **A3 ‚Äî Resolve time (syn re-parse).** Queue stores `syn::Type` (which is `Clone`) and resolver runs `render_type_inner(&ty)` on drain. Most composable ‚Äî preserves `render_type_string` as the fast exact-match path; adds `render_type_inner` as fallback only when exact match misses. Requires queue tuple shape change: `(String, String)` ‚Üí `(String, syn::Type)` or `(String, String, syn::Type)`.

  Facts: Option A2 requires reshaping `render_type_string` output, which would break the `signature_divergent` UDF invariant (`render_fn_signature` reuses `render_type` and must stay lexically canonical ‚Äî `type_render.rs:96-102`). Option A3 preserves the fast-path exact match on the rendered string and adds unwrap only on miss, matching the issue-body prescription "try render_type_string first, fall back to render_type_inner." Option A1 is symmetric but forces EVERY queue entry to pay the unwrap cost even when the outer render resolves directly.

**Ambiguity B ‚Äî How does the unwrapped-candidate matcher interact with the last-segment fallback?**
  The current resolver has two match tiers: (i) exact match in `emitted_item_qnames`, (ii) unique last-segment fallback. When `render_type_inner` produces a candidate like `"Foo"`, it could be:
  - B1: routed through BOTH tiers (exact `"Foo"` then last-segment `"Foo"`)
  - B2: routed through exact-match only (skip fallback because the unwrap is already more precise)
  - B3: routed through last-segment only (since the unwrapped inner name is typically short)

  Facts: the existing last-segment fallback is guarded against ambiguity (`by_last_segment.entry(seg).and_modify(|v| *v = None)`). Adding unwrapped candidates to the last-segment mix preserves that guard; there is no correctness risk either way, only a precision choice.

**Ambiguity C ‚Äî Does this issue touch `:Field.type_normalized` vs `type_path`?**
  RFC ¬ß3.5 line 143 says: "Both `type_normalized` and `type_path` are currently the same value ‚Ä¶ the split becomes meaningful once the `render_type_inner` follow-up (¬ß6) ships." The scar assertion at `tests/field_emission.rs:266-270` (`assert_eq!(tn, tp)`) would NOT flip if the issue only touches edge emission (which is the issue-body shape). It WOULD flip if the issue also changes `emit_field_list` to pass `render_type_inner(&f.ty)` as `type_normalized` and `render_type_string(&f.ty)` as `type_path`.

  Facts:
  - Issue #239 body does NOT mention `:Field.type_normalized` or `type_path`. Scope bullets are narrowly RETURNS/TYPE_OF edge emission.
  - The `:Field` attribute surface is frozen by Forbidden move #7 ("Do not reintroduce `type_qname` as a field property") ‚Äî this is about a different field, but signals caution.
  - RFC-037 ¬ß3.5 anticipates the split in THIS slice's timeframe but does not bind it to this issue specifically.
  - Splitting `type_normalized` from `type_path` semantically (unwrapped vs original) is a NEW schema meaning for the props. Even with wire shape unchanged, consumers of `:Field.type_normalized` would interpret it differently post-split.

  Decision surface for /prescribe: DEFER the field-prop split to a follow-up issue (safer, smaller PR), OR EXTEND this slice to flip the `type_normalized` semantic now (bigger PR, closes RFC ¬ß3.5 anticipation). Either is defensible.

**Ambiguity D ‚Äî Qualified wrapper paths (`std::vec::Vec<Foo>` vs `Vec<Foo>`)**
  `render_type_string` today produces path-segment-joined output: `Vec` for `Vec<T>` (single-segment), `std::vec::Vec` for `std::vec::Vec<T>` (3-segment), `alloc::vec::Vec` for `alloc::vec::Vec<T>`. When `render_type_inner` matches against the closed 9-wrapper list, does it match:
  - D1: only the bare last segment (`Vec`, `Option`, etc.) ‚Äî robust to `std::vec::Vec` / `alloc::vec::Vec` and aliases
  - D2: only the exact names (`Vec`, `Option`, etc.) ‚Äî misses qualified paths
  - D3: a small alias set per wrapper (e.g., `{ "Vec", "std::vec::Vec", "alloc::vec::Vec" }`)

  Facts: `syn::TypePath.path.segments` preserves source ordering. Matching on last-segment only (D1) aligns with how the existing last-segment fallback already handles ambiguity downstream. D2 silently misses `std::vec::Vec<Foo>` ‚Äî a real pattern in production code.

**Ambiguity E ‚Äî `Pin<Box<Foo>>` double-unwrap and the Ordering question**
  `Pin<P>` unwraps to `P`. If `P` is itself a wrapper (`Pin<Box<Foo>>`), does the depth counter see two wrapper layers (outer=Pin, inner=Box, leaf=Foo) or one (outer=Pin, inner-includes-Box-as-leaf)? The issue body says "Nested wrappers unwrap recursively with bounded depth" ‚Äî this is normative. Depth-counting must be consistent: is `Result<Vec<Foo>, Bar>` considered depth 2 (Result‚ÜíVec‚ÜíFoo) with two unwrap calls, or depth 3 (Result gives two candidates Vec<Foo> and Bar, each recurses)?

  Facts: Issue body example `Vec<Option<Arc<Foo>>>` is explicitly depth-3, terminating at `Foo`. This is consistent with "each unwrap step decrements the depth budget by 1, regardless of candidate count." `Result<Vec<Foo>, Bar>` at depth 3 would yield: outer=Result(2 candidates) ‚Üí inner=Vec<Foo>(depth 2 ‚Üí Foo at depth 1) AND inner=Bar(depth 2, terminal). Both `Foo` and `Bar` resolve. This is a prescription choice that affects test (b) `Result<Ok, Err>` ‚Üí two RETURNS edges. Prescription should confirm the depth-counting rule.
