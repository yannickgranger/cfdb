## Q1 — cfdb in /discover and /prescribe

### Bounded-context map of the actors

Before mapping verbs, the context boundaries must be stated or the mapping is groundless.

| Bounded context | Crates / artifacts it owns | What "discovery" means inside it |
|---|---|---|
| **cfdb** | cfdb-core, cfdb-extractor, cfdb-cli, cfdb-server, cfdb-hir-extractor, cfdb-store-lbug | Workspace-wide static code facts, persisted deterministically |
| **skill-layer (orchestration)** | `/discover`, `/prescribe`, `/boy-scout`, `/freshness`, `/work-issue`, `/operate-module` | Per-issue workflow artifacts: `.discovery/<issue>.md`, `.prescriptions/<issue>.md`, `.context/<issue>.md` |

These are genuinely different bounded contexts. The cfdb context's language centers on graphs, keyspaces, Cypher queries, enrichment passes, and deterministic snapshots. The skill-layer context's language centers on issues, sessions, gates, prescriptions, and coding agents. **No term means the same thing in both.** (See homonym report below.)

### /discover wiring — section to cfdb verb mapping

The `/discover` skill currently performs these analytical sections via grep/read/bash. Each section maps to one or more cfdb queries:

| /discover section | Current mechanism | cfdb verb(s) that replace/augment it | Notes |
|---|---|---|---|
| **Concept Inventory Step 1a** — find all impls of a concept name | `Grep` for `struct`, `trait`, `enum`, `fn` | `find_canonical(keyspace, concept)` + `query_raw` over `:Item{name: contains concept}` | cfdb gives workspace-wide recall; grep misses macro-generated items (§13 Risk 2) |
| **Step 1b** — classify each match (PORT/DOMAIN/ADAPTER etc.) | Read file, inspect crate path | `query_raw` on `:Item.crate` + `(:Crate)-[:BELONGS_TO]->(:Context)` from `enrich_bounded_context` | Context label from `enrich_bounded_context` maps directly to the PORT/DOMAIN/ADAPTER classification |
| **Step 1d** — Global Resolution Census (FromStr/Display audit) | `rg "impl FromStr for <Type>"` | `query_raw` on `(:Item{kind:'Impl'})-[:IMPLEMENTS]->(:Item{name:'FromStr'})` WHERE `RETURNS` type matches | Gives workspace-wide census; grep misses impl blocks generated by macros |
| **Step 1e** — Const table ancestry | `rg "const <NAME>"` across workspace | `query_raw` on `:Item{kind:'Const', name: contains concept}` — all crates simultaneously | Detects >50% overlap via `signature_hash` similarity (UDF) rather than string sampling |
| **Step 1f** — Split-brain pre-scan | `audit-split-brain --full-workspace` binary | `list_violations(keyspace, "split-brain.cypher")` + `diff` against prior snapshot | cfdb replaces the binary scan; `diff` surfaces NEW splits introduced by the branch |
| **Step 1g** — MCP/CLI adapter scope grep | `rg "<TypeName>" crates/qbot-mcp/src/` | `query_raw` filtering `:Item` in crates where `bounded_context = 'mcp'` that reference the type | Needs `enrich_bounded_context` pass to be complete |
| **Step 2a** — Identify entry points | Manual grep in qbot-mcp, *-cli | `query_raw` on `:EntryPoint` nodes filtered by `crate` or `kind` | Requires v0.2 `:EntryPoint` catalog |
| **Step 2b** — Data Flow Trace (param flow) | Manual BFS reading files | `query_raw` BFS over `CALLS*` from `:EntryPoint`, filtering for param-resolution `:Item`s | Requires `CALLS` edges (v0.2 HIR extractor) |
| **Step 2c** — Caller Census | `grep -rn "<fn>(" crates/*/src/` | `list_callers(keyspace, qname)` | Direct typed verb — no Cypher authoring needed |
| **Step 3** — Decorator chains | `rg "impl <TraitName> for"` + manual | `query_raw` for `(:Item)-[:IMPLEMENTS]->(:Item{name:trait})` WHERE `:Item` wraps another impl | Pattern only detectable structurally; requires IMPLEMENTS_FOR edges |
| **Step 4** — Ownership Map (port fan-out) | Count callers via grep | `list_callers` per port method; count > 1 = split risk | Simple |
| **Step 6a** — Decision Archaeology | `git log --follow -L` | `enrich_history` provides `:Item.git_last_author`, `:Item.git_commit_count`; RFC references via `enrich_rfc_docs` → `(:Item)-[:REFERENCED_BY]->(:RfcDoc)` | Git blame is still required for verbatim commit messages; cfdb provides the structural signal, not the prose |

**Gaps — verbs or schema nodes missing:**

1. **Missing verb: `list_definitions_of(keyspace, concept_name)`** — a convenience verb returning all `:Item` nodes where `name` matches the concept name, grouped by bounded context. Currently requires hand-crafting a `query_raw` call. This is the single most frequent call pattern in Steps 1a/1b and the most error-prone for a skill agent authoring Cypher. Justification: it unblocks the Concept Inventory in one typed call without requiring the skill to know the Cypher label vocabulary.

2. **Missing verb: `list_entry_points(keyspace, kind_filter?)`** — returns `:EntryPoint` nodes, optionally filtered by kind (`mcp_tool`, `cli_command`, etc.). This is the starting node for Step 2a and Step 2b. Currently /discover must hard-code grep paths (`crates/qbot-mcp/src/`, `crates/*-cli/src/`). A typed verb removes the hard-coded crate-path assumption.

3. **Schema gap: bounded-context assignment for items at crate-prefix boundary** — `enrich_bounded_context` uses the crate-prefix convention but cross-cutting crates (`messenger`, `sizer`, `executor`) require `.cfdb/concepts/*.toml` override. The schema needs a `Context` node type with `(:Crate)-[:BELONGS_TO]->(:Context)` edges that `enrich_bounded_context` populates. v0.2 addendum §A2.2 already describes this but it is not in v0.1 §7 schema. This edge is load-bearing for Steps 1b and 1g. **Extension proposal**: add `:Context {name, canonical_crate, owning_rfc}` node and `BELONGS_TO` edge to the schema at v0.1 minor bump; determinism impact = additive (G3 compliant); syn cost = zero (derived from Cargo.toml metadata + `.cfdb/concepts/*.toml`).

4. **Schema gap for Decision Archaeology**: `enrich_rfc_docs` produces `(:Item)-[:REFERENCED_BY]->(:RfcDoc)` but the RFC node type is not listed in §7. The discovery artifact needs to know which `:Item`s are mentioned in which RFCs and what scope verdict the RFC assigns (deferred, current, blocked). Extension: add `:RfcDoc {path, scope_verdict}` node with `REFERENCED_BY` edge. Determinism impact = additive; cost = doc-file parsing, no compiler dependency.

**Determinism impact of proposals:** both proposals are additive enrichments (G3). Neither modifies structural facts. Both are syn-cost only. Open-newtype encoding (#3670) applies — `EdgeLabel::BELONGS_TO` and node label `"CONTEXT"` as `pub const`.

| /prescribe step | Current mechanism | cfdb verb(s) | Notes |
|---|---|---|---|
| **Step 3 Decision Tree** — concept exists? | Reads `.discovery/<issue>.md` concept inventory | `find_canonical(keyspace, concept)` returns the same data as the discovery section in structured form | cfdb is the oracle; discovery section becomes a cached view |
| **Step 5b** — Independent Resolution Census (CREATE anti-bias) | `rg "impl FromStr for <Type>"` in prescribe agent itself | `query_raw` on `IMPLEMENTS` graph filtered by type name + `FROM_STR` label | **This is the split-brain test**: "two resolvers = one future bug". cfdb makes this workspace-wide and recall-complete |

| **Step 5e** — Market-Phenomenon Heuristic | Hard-coded `domain-market-structure` grep | `query_raw` WHERE `Item.crate = 'domain-market-structure'` AND name matches heuristic | cfdb makes the crate boundary explicit and the query precise |
| **Step 5f** — MCP/CLI EXTEND Scan | `rg "<TypeName>" crates/qbot-mcp/src/` | `query_raw` on `:Item` in `bounded_context = 'mcp'` that `CALLS` or `TYPE_OF` the changed type | Needs `CALLS` for full accuracy; partial result available from `TYPE_OF` in v0.1 |
| **Canonical Ownership Assertion** | Prose in prescription artifact | `find_canonical(keyspace, concept)` confirms the canonical `:Item`; `list_bypasses(keyspace, concept)` surfaces bypass callers | These two typed verbs together answer "is there a canonical and who bypasses it?" — directly answering the split-brain test |

**Verify each prescription anti-split-brain check has a cfdb query backing it:**

- Step 5b (Resolution Census): `query_raw` on IMPLEMENTS + type name → YES, backed
- Step 5c (Test-Parser Divergence): needs `Item.is_test` attribute — **CURRENTLY MISSING**
- Step 5d (Name Collision): `query_raw` on Item.name exact match → YES, backed
- Step 5e (Market-Phenomenon): `query_raw` on Item.crate filter → YES, backed
- Step 5f (MCP/CLI EXTEND): partially backed in v0.1 via TYPE_OF; fully backed in v0.2 via CALLS

**Additional gap: `list_context_owner(keyspace, concept)` verb** — the prescription bounded-context test (parent §4) asks "which context OWNS this concept, not just which USES it?" The current `find_canonical` verb returns items by concept name but does not distinguish owning context from consuming context. This is load-bearing for the step 5d branch "same concept, different context → justify CREATE". The prescription cannot make that judgment without knowing which context owns the canonical. **Extension proposal**: add `list_context_owner(keyspace, concept)` returning `{owning_context, canonical_qname, consuming_contexts[]}`. Internally this is a `query_raw` over `CANONICAL_FOR` + `BELONGS_TO` edges. Expose as a typed verb because it is called on every CREATE decision. Determinism impact = zero (read-only query over existing facts). Schema cost = `CANONICAL_FOR` edge is already in §7.

### Verdict on Q1

`YELLOW` — the 15 shipped verbs cover the majority of /discover and /prescribe queries, but three specific gaps block full wiring: (1) `Item.is_test` attribute missing (needed for Step 5c test-parser divergence check), (2) `BELONGS_TO` edge between `:Crate` and `:Context` node not in v0.1 schema (needed for Steps 1b, 1g, and context-ownership assertions), and (3) `list_context_owner` typed verb absent (needed for the bounded-context test on every CREATE decision in /prescribe). None requires breaking changes; all are additive schema extensions and one new typed verb.

## Q2 — Permanent watchdog

### Tier table

| Tier | Trigger | Runs | Writes | Blocks |
|---|---|---|---|---|
| **Per file-save (IDE)** | File-change event (lefthook / watchman) | `cfdb query_raw` on changed file's containing crate only (not full extract) — returns violations touching that file | Nothing persisted; result shown inline | No — advisory only |
| **Per session** | `/work-issue` Phase 0 invokes `/freshness`, which calls `/gate-freshness` | `/freshness` checks if `.cfdb/inventory.json` SHA matches current `origin/develop` HEAD; if stale, emits staleness warning in context package | Nothing new; marks inventory staleness flag in `.context/<issue>.md` | YES — `/discover` hard-blocks on `verdict=abort` if staleness > N commits (proposed N=50) |
| **Per session** | `/discover` Step 1f (split-brain pre-scan) | `list_violations(keyspace, "split-brain.cypher")` filtered to issue scope | Appended to `.discovery/<issue>.md` as EXISTING RESOLUTION POINTS section | No — facts only; `/prescribe` decides |
| **Per PR (CI)** | PR opened or commit pushed | `cfdb diff keyspace_base keyspace_branch --kinds=Item,CallSite` → new violations since base | Diff output to PR comment; violation report to `.cfdb/pr-<pr>.json` | YES — new CFn-class findings (context_homonym, canonical_bypass) block merge; new duplicate/unwired findings warn only |
| **Nightly** | Cron `0 2 * * *` | `cfdb extract` against develop HEAD; re-run all `.cypher` rules; refresh `.concept-graph/RESCUE-STATUS.md` counts | Updated `.cfdb/inventory.json`, updated `RESCUE-STATUS.md` with counts and operate-module candidates | No — generates signals for next session bootstrap |
| **Weekly** | Cron `0 3 * * 1` | `cfdb diff` between this week's snapshot and last week's; emit `RESCUE-STATUS.md` delta section | `.concept-graph/RESCUE-STATUS-delta-<date>.md` | No |

### Inventory lifecycle

**Where it lives:** `.cfdb/inventory.json` in-repo (as a committed file, per Q12 vote (a) in v0.2 §A5), refreshed by nightly CI commit. Keyspace snapshots (the lbug graph files) live in `.cfdb/snapshots/` — rebuildable cache, NOT committed to git. Only the JSONL canonical dump (deterministic) and `inventory.json` (staleness index) are committed.

**Who refreshes it:** Nightly cron running `cfdb extract` against develop HEAD. On a schema version bump, CI also runs the refresh inline. No session agent ever runs `cfdb extract` — that is the cron's job. A session that finds a stale inventory reads the last committed state (with a warning), never blocks on extraction.

**Staleness detection:** `/freshness` skill Step 2e reads `.cfdb/inventory.json` frontmatter field `extracted_at_sha`. It compares against current `git rev-parse origin/develop`. If the delta exceeds N commits (proposed 50 — calibrate from v0.2 telemetry), it sets `inventory_stale=true` in the context package. `/discover` reads this flag and emits a staleness warning in its artifact but does NOT refuse to run — stale facts are still better than no facts. Only a complete absence of the inventory file causes a HARD BLOCK.

**Invalidation contract with /freshness:** `/freshness` is the staleness detector, not the refresher. It reports the flag into the context package. The session agent cannot refresh the inventory (that would require running `cfdb extract` which has 2–4 GB peak RSS per v0.2 §A1.2). The session proceeds with acknowledged staleness. The nightly cron resolves it overnight.

**DDD note on context mapping between /freshness and cfdb**: the relationship is **Conformist**. `/freshness` consumes cfdb's inventory facts (reads them as-is) without translation. `/freshness` does not define what a "finding" is — it reads cfdb's `:Finding` schema directly. This is load-bearing: if `/freshness` translated cfdb facts, it would own a parallel vocabulary that would drift from cfdb's as cfdb evolves.

### CI failure modes

**What makes a PR fail (block):**
- `cfdb diff` finds a new `:Finding` of class `context_homonym` introduced by the PR — a new homonym crosses context boundaries. These require architect judgment (ACL, Shared Kernel, etc.) and cannot be auto-fixed by a sweep. Block + route to `/operate-module`.
- `cfdb diff` finds a new `canonical_bypass` finding — a new call site goes around the declared canonical. Block + route to `/sweep-epic` (rewire class).
- Any existing `.cypher` ban rule (Pattern D/E/F) returns non-empty results for code changed by the PR. Block + emit violation details in PR comment.

**What makes a PR warn (non-blocking):**
- New `duplicated_feature` finding — same-context duplicate. Warning + route to `/sweep-epic` (consolidate). Non-blocking because this class was explicitly approved for gradual reduction per v0.2 §A4 doctrine bootstrap.
- New `random_scattering` finding — Warning + route to `/boy-scout` for inline fix. Non-blocking.
- New `unwired` finding — Warning only. Requires human review to distinguish "dead code" from "code awaiting wiring".

**How auto-remediation routes:**

The `SkillRoutingTable` (`.cfdb/skill-routing.toml` per v0.2 §A2.3) maps class to skill at CI message time:

| Class | CI action | Routed skill |
|---|---|---|
| `context_homonym` | Block + tag PR `needs-council` | `/operate-module` (architect decision required) |
| `canonical_bypass` | Block + tag PR `needs-rewire` | `/sweep-epic` |
| `duplicated_feature` | Warn | `/sweep-epic` |
| `random_scattering` | Warn | `/boy-scout` |
| `unfinished_refactor` | Warn | `/sweep-epic --mode=port` |
| `unwired` | Warn | `/boy-scout` delete OR issue-owner session |

**DDD-critical point**: `context_homonym` is the ONLY class that blocks unconditionally. All other classes have non-blocking remediation paths. This is correct: a homonym is a boundary violation that requires a Context Mapping decision — no tool can auto-remediate it. A CI gate that blocks on homonyms but only warns on duplicates correctly reflects the DDD priority ordering.

### Verdict on Q2

`GREEN` — the tier model is coherent, the inventory lifecycle has a defined invalidation contract with /freshness, and the CI failure mode table correctly distinguishes homonym (block, requires council) from mechanical debt (warn, auto-routed to sweep/boy-scout). No blocking gaps. The staleness threshold (N=50 commits) needs empirical calibration from v0.2 — that is acceptable for v0.2 gate item scope.

## Q3 — Missing skills

### /operate-module

**Description:** Evaluate cfdb infection inventory against §A3.2 thresholds for a bounded context; emit raid plan markdown if threshold crossed; otherwise route to /boy-scout.

**Arguments:** `<context-name> <inventory-json-path> [--workspace <path>]`

**Inputs:**
- `<inventory-json-path>` — structured JSON from `cfdb query_raw` scoped to the named context (§A3.3 shape: `findings_by_class`, `canonical_candidates`, `reachability_map`, `loc_per_crate`)
- `.context/<context-name>.toml` (optional) — architect overrides declaring the context's canonical crate and owning RFC
- The `SkillRoutingTable` at `.cfdb/skill-routing.toml`

**Outputs:**
- If threshold crossed: `raid-plan-<context>.md` at workspace root, with `council_required=true` marker
- If below threshold: routing verdict JSON `{ "verdict": "below_threshold", "route_to": "/boy-scout", "findings": [...] }` written to stdout for the orchestrator to consume

**Invariants (what it must NEVER do):**
- NEVER run `cfdb extract` — it receives a pre-built inventory
- NEVER execute code changes — read-only planning
- NEVER declare a context-mapping resolution (ACL, Shared Kernel, etc.) — that is council territory. It raises the question and emits it in the `## Council decisions required` section of the raid plan
- NEVER route `context_homonym` findings to `/boy-scout` — homonyms require council, always

**Invocation protocol (how /work-issue decides to invoke):**
`/work-issue` does NOT invoke `/operate-module` directly. `/operate-module` is invoked by the **main agent** (outside the standard work-issue pipeline) when:
1. The CI drift gate reports `context_homonym` or block-level findings for a bounded context
2. The nightly cron update to `RESCUE-STATUS.md` shows a context crossing threshold
3. The user explicitly invokes it to scope a raid

It is NOT part of the per-issue pipeline. It is a **per-context** planning tool triggered by threshold events.

**Relationship to other skills:**
- **Before /operate-module**: `cfdb extract` (cron) → produces inventory; `/freshness` (session) → confirms context package is current
- **/operate-module** → produces raid plan
- **After /operate-module**: COUNCIL REQUIRED (human + architect review) → approved RFC → `/sweep-epic --mode=port --raid-plan=<path>` executes portage
- **/discover** and **/prescribe**: operate-module does NOT invoke these. After the raid plan is approved, each sub-issue in the resulting EPIC goes through the full `/work-issue` pipeline (including /discover and /prescribe) individually

**Failure modes and escape hatches:**
- `inventory-json-path` file not found → HARD BLOCK. Emit "inventory not found — run `cfdb extract` first"
- `context-name` not found in `enrich_bounded_context` output → HARD BLOCK. Emit "context '<name>' not in cfdb graph — check `.cfdb/concepts/<name>.toml` or verify crate-prefix convention"
- Threshold crossed for multiple contexts simultaneously → emit one raid plan per context, each as a separate file. The orchestrator (human or main agent) decides sequencing
- cfdb data is stale (inventory older than N commits) → emit staleness warning in raid plan header; do NOT refuse to run (stale data is still useful for planning)

### /gate-raid-plan

**Description:** Validate a submitted raid plan against the cfdb fact base using Pattern I Cypher queries; block if hidden callers or dangling references exist.

**Arguments:** `<raid-plan-path> <context-name> [--workspace <path>]`

**Inputs:**
- `<raid-plan-path>` — the markdown raid plan doc produced by `/operate-module` (contains portage, rewrite, glue, drop lists in structured `##` sections)
- The cfdb keyspace for the workspace (read via `cfdb query_with_input`)
- The 5 Pattern I Cypher files from `examples/queries/raid-validate-*.cypher`

**What it asserts (per §3.9 5-query composition):**
1. **Completeness** — every item in the portage list EXISTS as an `:Item` in cfdb. Missing items = extraction failure or stale plan.
2. **Dangling-drop** — no item in the "dead list" (drop bucket) has callers in any non-drop bucket. Dangling drop = would break callers.
3. **Hidden callers** — no item in portage/rewrite buckets has callers OUTSIDE the context boundary that are NOT in the misplaced list. Hidden callers = raid breaks external consumers.
4. **Missing canonical** — if the raid plan declares a canonical item, cfdb confirms it exists and has `fan_in > 0` from within-context call sites. Zero fan-in = dead canonical.
5. **Clean/dirty-mismatch** — items in the portage list must not already exist at the declared "new home" crate (would create duplicate). Items in the drop list must not be referenced by items in the portage list.

**Returns:**
- If all 5 pass: `{ "verdict": "PASS", "findings": [] }` — raid plan is structurally sound
- If any fail: `{ "verdict": "BLOCK", "findings": [ {"query": N, "violation": ..., "evidence": file:line} ] }` — raid cannot proceed

**Where in the pipeline it runs:** after `/operate-module` emits the draft raid plan AND after council review, BEFORE `/sweep-epic --mode=port` starts. The gate is the mechanical verification step that replaces human checking of "did we miss any callers?" Sequence: `/operate-module` → council approves → `/gate-raid-plan` → if PASS → `/sweep-epic --mode=port`.

**Failure modes:**
- cfdb graph is stale relative to the raid plan's target branch → run `cfdb diff` between the plan's pinned SHA and current HEAD first. If delta includes items in the portage list, re-run `/operate-module` against the updated inventory before re-gating.
- Raid plan markdown has malformed section headers (portage/dead/misplaced lists not parseable) → BLOCK with parse error. The `/operate-module` output template must produce machine-parseable `##` sections.

### /cfdb-scope

**Resolution: CLI flag, not a skill.** Justify:

A skill exists because it has orchestration logic, gate interactions, or artifact production that requires a multi-step agent. `/cfdb-scope` is none of those — it is a scoped query against the cfdb graph that returns structured JSON. Adding a skill for this would violate the parent RFC §4 invariant ("cfdb is not opinionated about workflows") and the BRIEF's SOLID brief (ISP — consumers should depend only on the slice they query).

**Argument shape:**
```
cfdb scope --workspace <path> --context <context-name> [--format json|table] [--keyspace <sha>]
```

**CLI semantics:**
- `--context <context-name>` — filter all output to `:Item`s belonging to the named bounded context (via `BELONGS_TO` edge or crate-prefix heuristic)
- `--format json` — emits the §A3.3 structured inventory JSON, suitable for piping directly to `/operate-module`
- `--keyspace <sha>` — use a specific snapshot rather than latest. Defaults to latest available
- Without `--context`: error. "Context name is required. To list known contexts: `cfdb scope --list-contexts`"

**How it fits in the invocation flow (§A3.4):**
```
cfdb scope --workspace . --context trading --format json > /tmp/trading-inventory.json
/operate-module trading /tmp/trading-inventory.json --workspace .
```

This is a pipe, not a skill chain. The inventory JSON is a first-class intermediate artifact that any agent or human can inspect. The absence of a skill boundary here is intentional — keeping it a CLI flag means cfdb stays workflow-agnostic.

**DDD note on the Published Language role:** `cfdb scope`'s JSON output is the **Published Language** boundary between cfdb and the skill layer. cfdb owns the schema (`findings_by_class`, `canonical_candidates`, etc.); the skill layer consumes it conformantly. No ACL translation is needed because cfdb and the skill layer share this schema as the explicit contract. If cfdb's output schema changes, the skill layer must update — that is the Conformist relationship for consumers and the Published Language obligation for cfdb.

### /boy-scout --from-inventory

**Resolution: single skill, two modes.** Justify:

A sibling skill would duplicate the entire scan-select-fix-verify-commit loop. The difference between existing boy-scout (file-proximity triage) and inventory-driven triage is only in **Step 2 (how violations are obtained)** and **Step 3 (how the batch is selected)**. All other steps (fix, compile, commit, report) are identical. A second mode flag `--from-inventory <path>` is the ISP-minimal change.

**Mode comparison:**

| Aspect | Existing mode (proximity) | New mode (--from-inventory) |
|---|---|---|
| Violation source | `boy-scout-scope` binary (file proximity, 2 hops) | cfdb inventory JSON scoped to issue context |
| Batch selection | 50% of proximity violations, capped at 5 | 50% of inventory findings in class {`random_scattering`, `unwired`}, capped at 5 |
| Fix type | Mechanical: bare .unwrap(), generic .expect(), bare #[ignore] | Same mechanical violations PLUS: extract helper for `random_scattering` if pure text-edit, delete for `unwired` with no tracker |
| Budget source | `boy-scout-scope` JSON `budget` field | `count(inventory.findings_by_class.random_scattering) + count(inventory.findings_by_class.unwired)` × 0.5 |

**Class filter — which debt classes does boy-scout own:**

| Class | Boy-scout owns? | Justification |
|---|---|---|
| `random_scattering` | YES | Mechanical extract-helper; no bounded-context decision required; inline fixable |
| `unwired` (no tracker) | YES — delete only | Mechanical delete; no architecture judgment |
| `unwired` (has tracker) | NO — route to issue owner | Requires judgment about whether to wire or delete |
| `duplicated_feature` | NO | Requires picking "head to keep" — architecture decision, not mechanical |
| `unfinished_refactor` | NO | Requires completing a migration — scope exceeds inline fix |
| `canonical_bypass` | NO | Requires rewire through the canonical — call-graph change, exceeds text-only mechanical fix |
| `context_homonym` | ABSOLUTELY NOT — v0.2 Q10 rationale | Homonym is a Context Mapping decision. Boy-scout has no authority to pick ACL vs Shared Kernel. Routing a homonym to boy-scout is the textbook split-brain-by-cleanup scenario |

**DDD critical: v0.2 Q10 says `context_homonym` is always `/operate-module`**. The reasoning: a homonym is a domain language ambiguity, not a code style violation. The remedy is a DDD decision (which context owns the concept, what translation layer separates them). Boy-scout's authority is mechanical fixes within a doughnut. It has no authority to declare "context X owns this, context Y should use an ACL" — that is an architectural, linguistic decision that must survive through multiple sessions and be recorded in a raid plan. A boy-scout commit that silently renames a homonym resolves the name collision but not the semantic collision, and leaves the next session blind to the original DDD problem.

**Input contract for --from-inventory:**
```json
{
  "context": "trading",
  "findings_by_class": {
    "random_scattering": [
      { "item_a": "qname_a", "item_b": "qname_b", "file_a": "...", "line_a": 42, "file_b": "...", "line_b": 67, "evidence": "near-identical AST body" }

      { "item": "qname_c", "file": "...", "line": 99, "tracker": null }
    ]

```
The `findings_by_class` object is a strict subset of the §A3.3 inventory — only the two classes boy-scout owns. The orchestrator (main agent or CI) filters to these two classes before handing the inventory to boy-scout. This preserves ISP on the boy-scout side: it never sees context_homonym findings and cannot be confused into acting on them.
