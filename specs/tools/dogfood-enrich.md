# Spec: dogfood-enrich

The RFC-039 §3.5.1 self-dogfood harness binary — `dogfood-enrich` reads a `.cfdb/queries/self-enrich-<pass>.cypher` template, substitutes the threshold const for ratio passes, invokes `cfdb violations` against the materialized tempfile, and maps the row-count outcome to a 0 / 30 / 1 exit code. Lives under `tools/` (not `crates/`) so the leaf binary's CI-policy thresholds remain `Ca = 0` per the RFC-039 R2 SAP analysis (consts that govern CI gates must not couple to `cfdb-cli`'s efferent stability profile, and must not contaminate `cfdb-core`'s inner-ring zero-dependency invariant). Consumed by `.gitea/workflows/ci.yml` for the four default-feature dogfoods (`enrich-deprecation`, `enrich-rfc-docs`, `enrich-bounded-context`, `enrich-concepts`) and by the future nightly workflow for the three feature-gated passes.

Bounded-context note: this tool is NOT a cfdb library crate. It ships as an independent binary + library (`dogfood-enrich` crate at `tools/dogfood-enrich/`), depends on `cfdb-core` only for the wire-form `EnrichReport` type, and explicitly does NOT link `cfdb-cli` — it invokes `cfdb` as a subprocess, mirroring the relationship `ci/cross-dogfood.sh` already uses. The subprocess-only design is the resolution to the RFC-039 R2 council split between clean-arch (Dependency Rule, prefer Option β) and solid-architect (CCP, prefer Option α): a standalone binary that does NOT couple compile-time to `cfdb-cli` satisfies both lenses simultaneously.

## FeatureGate

The Cargo feature flag a pass requires before its sentinel can be evaluated — `Default` (PR-time eligible, no flag), `Hir` (`--features hir`, requires HIR-extracted `:EntryPoint` nodes for reachability), `QualityMetrics` (`--features quality-metrics`, requires syn re-parse for `cyclomatic` + `unwrap_count`), and `GitEnrich` (`--features git-enrich`, requires libgit2 against the workspace `.git`). The non-`Default` variants gate the corresponding pass to the parallel nightly job per RFC-039 §3.3 — building `cfdb-cli` + `dogfood-enrich` once with all three flags enabled is cheaper than three per-feature jobs (rust-systems R2 verdict: the 31 `ra_ap_*` crates dominate compile time and are shared across features). The harness consults `PassDef.feature_required` together with the I5.1 [`feature_guard::check_pass_ran`] guard before invoking the violations sentinel; off-feature builds short-circuit with `GuardError::FeatureMissing` rather than silently reporting 100% null coverage. Defined at `tools/dogfood-enrich/src/passes.rs:13`.

## PassDef

Static descriptor for one of the 7 RFC-039 enrichment passes — `{ name: &'static str, query_template_path: &'static str, threshold: Option<u32>, feature_required: FeatureGate }`. The static catalogue lives at [`PassDef::all`] (`tools/dogfood-enrich/src/passes.rs:48`) and matches the order of the §3.1 invariant table. `name` is the canonical identifier accepted by `cfdb enrich-<name>` and `cfdb violations` rule lookup — stable; renaming requires bumping every `.cfdb/queries/self-enrich-*.cypher` cross-reference. `query_template_path` is the workspace-relative location of the Cypher template (templates added by Issues #343–#349). `threshold` is `Some(N)` for the four ratio passes (`enrich-bounded-context`, `enrich-reachability`, `enrich-metrics`, `enrich-git-history`) sourced from `crate::thresholds::*`, and `None` for the three hard-equality passes (`enrich-deprecation`, `enrich-rfc-docs`, `enrich-concepts`) — the harness substitutes `{{ threshold }}` only for `Some` variants. `feature_required` selects the [`FeatureGate`](#featuregate) used by I5.1. `PassDef::by_name` provides the runtime lookup used by `main.rs` to dispatch on `--pass`. Defined at `tools/dogfood-enrich/src/passes.rs:29`.

## RunOutcome

Successful outcome of the violations subprocess — `Clean` when `cfdb violations --count-only --no-fail` returns zero rows on stdout, `Violations { row_count: i32 }` when ≥1 row is returned. Maps in [`main.rs`](#main) to `EXIT_OK` (0) and `EXIT_VIOLATIONS` (30) respectively. Any subprocess failure (signal, unexpected exit, unparseable count) is bubbled as a [`RunnerError`](#runnererror) rather than being re-classified as `Violations` — the violation outcome is reserved for the well-defined "rule fired ≥1 row" path. Defined at `tools/dogfood-enrich/src/runner.rs:78`.

## RunnerError

Errors that bubble up from the runner to the binary's `EXIT_RUNTIME_ERROR` (1) exit code via `thiserror::Error`. Six variants: `TemplateMissing(PathBuf)` for an absent `.cypher` template (the scaffolding-stage scenario before Issues #343–#349 ship), `TemplateRead { path, source: io::Error }` for read-failures, `TempfileWrite { path, source: io::Error }` for materialization failures, `SubprocessSpawn { binary, source: io::Error }` for spawn failures, `SubprocessSignal { binary }` for subprocesses terminated without an exit code, `SubprocessUnexpectedExit { binary, exit, stderr }` for non-zero exits with `--no-fail` (which should never happen unless `cfdb` itself errored), and `CountParse { binary, stdout }` for unparseable row counts on stdout. The split between `SubprocessSpawn`, `SubprocessUnexpectedExit`, and `SubprocessSignal` distinguishes pre-launch failures from post-launch failures — useful when diagnosing CI flakes from the captured stderr. Defined at `tools/dogfood-enrich/src/runner.rs:36`.

## ConceptCounts

TOML-scan result for the `enrich-concepts` source-side ground truth — `{ distinct_context_names: usize, declared_canonical_crate_count: usize }`. Returned by `scan_concepts::scan_concepts` (text-only TOML walk over `<workspace>/.cfdb/concepts/*.toml`, stdlib-only, no `toml` crate dep). The two fields drive the `{{ declared_context_count }}` and `{{ declared_canonical_crate_count }}` substitutions in `.cfdb/queries/self-enrich-concepts.cypher` per RFC-039 §3.1.2 — the first is the dedup-by-name distinct count of context entries (TOMLs with the same top-level `name = "X"` collapse), the second is the count of TOMLs declaring a non-empty `canonical_crate = "..."` value. The conditional sentinel (c) — `IF declared_canonical_crate_count > 0 THEN count(:CANONICAL_FOR) > 0` — short-circuits when the count is zero, mirroring the `Option<String>` shape of `cfdb_concepts::ContextMeta::canonical_crate` (`crates/cfdb-concepts/src/lib.rs:65`). Defined at `tools/dogfood-enrich/src/scan_concepts.rs:72`.

## GuardError

I5.1 feature-presence guard error type — five variants surfaced by `feature_guard::check_pass_ran`: `Spawn { binary, source }` for `cfdb` invocation failures, `Signal { binary }` for signal-terminated subprocesses, `NonZeroExit { binary, exit, stderr }` for `cfdb enrich-<pass>` exiting non-zero (a real cfdb error, not feature-missing), `JsonParse { binary, stdout, source }` when the `EnrichReport` envelope can't be deserialized, and `FeatureMissing { pass, warnings }` when the pass dispatched to its off-feature path (`crates/cfdb-petgraph/src/enrich_backend.rs:178-262`) and reported `ran: false`. The latter is the load-bearing variant — the harness MUST exit before invoking the violations sentinel because a feature-absent build would silently report 100% null coverage and look like a real regression (RFC-039 §I5.1 directive). The error message embeds the warnings field from the `EnrichReport` JSON envelope so the user-facing diagnostic names the missing feature flag (`hir`, `quality-metrics`, or `git-enrich`). Defined at `tools/dogfood-enrich/src/feature_guard.rs:20`.
