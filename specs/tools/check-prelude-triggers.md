# Spec: check-prelude-triggers

The Tier-1 mechanical pre-council trigger binary ratified in RFC-034 v3.3 §4.2 — `check-prelude-triggers` evaluates the five deterministic C-triggers (`C1` cross-context, `C3` port-signature, `C7` financial-precision, `C8` pipeline-stage, `C9` workspace-cardinality) against a git diff snapshot and emits a frozen-schema JSON envelope on stdout. Lives under `tools/` (not `crates/`) so the load-independent floor can run in consumer repos that don't vendor cfdb's library surface. Consumed by `/freshness` Step 2g which merges per-trigger envelopes into `.triggers/<issue>.json`.

Bounded-context note: this tool is NOT a cfdb library crate. It ships as an independent binary + library (`check-prelude-triggers` crate at `tools/check-prelude-triggers/`), deliberately avoids a dependency on `cfdb-concepts` or any other cfdb crate (see the Duplication note on `LoadError` below), and is anti-drift-gated under the second `--code tools/` CI pass introduced in issue #137.

## PreludeTriggerReport

The frozen RFC-034 §4.2 envelope serialized to stdout per invocation — fields `schema_version` (always `"v1"`, frozen via the `pub const SCHEMA_VERSION: &str = "v1"` sentinel at `tools/check-prelude-triggers/src/report.rs:28`), `from_ref`, `to_ref`, `triggers_fired: Vec<TriggerId>` (sorted, deduped list of fired C-trigger IDs), and `evidence: BTreeMap<String, serde_json::Value>` (per-trigger evidence payload keyed by the trigger's `as_str()` form, e.g. `"C1"`). Handlers call `record(id, evidence)` to append a firing; consumers (`/freshness` Step 2g) read the deserialized envelope from `.triggers/<issue>.json` and merge the five per-trigger envelopes using a union on `triggers_fired` and a shallow merge on `evidence`. Bumping `SCHEMA_VERSION` is a BREAKING change every consumer skill must opt into; additive trigger changes (new variants in [`TriggerId`](#triggerid)) keep the value at `"v1"` per the OCP contract in RFC-034 v3.3 §4.2. Defined at `tools/check-prelude-triggers/src/report.rs:32`.

## TriggerId

Closed enum of the five RFC-034 mechanical C-triggers — `C1`, `C3`, `C7`, `C8`, `C9`. Serializes to the exact RFC-034 wire spelling via `#[serde(rename = "C1")]` per variant; `as_str(self)` returns the same uppercase string form for diagnostics. Additive (OCP): future Tier-2 promotions append new variants without breaking consumers that parse the string. Output-only — this enum has no `FromStr` and no paired parse-error type because the tool writes envelopes and never parses them. Defined at `tools/check-prelude-triggers/src/trigger_id.rs:11`.

**Homonym note.** This `TriggerId` is the RFC-034 mechanical pre-council trigger enum (`C1`, `C3`, `C7`, `C8`, `C9`). It is a DDD homonym with `cfdb_cli::check::TriggerId` — the editorial-drift `T1`/`T3` enum spec'd at [`specs/concepts/cfdb-cli.md`](../concepts/cfdb-cli.md). The two enums live in different bounded contexts and evolve on independent change vectors; they share only a lexical name, and each carries its own variants, its own serialization rules (this one is Serialize-only; the `cfdb-cli` homonym has a paired `FromStr` + `UnknownTriggerId`), and its own ownership lineage. The deliberate homonym is documented at `tools/check-prelude-triggers/src/lib.rs:18-21`; `audit-split-brain` flags the duplicate type name and confirms both definitions are legitimate per their respective bounded contexts.

## TriggerOutcome

Per-trigger evaluation result returned by each handler's `evaluate*` fn — `{ fired: bool, evidence: serde_json::Value }`. A `fired: true` outcome means pre-council review is MANDATED for the diff under evaluation; the `evidence` payload is trigger-specific (e.g. `contexts_touched: [...]` for C1, `matched_paths: [...]` for C3) and gets embedded in the outer [`PreludeTriggerReport`](#preludetriggerreport)'s `evidence` map under the trigger's string key. Internal to the binary's dispatch; not consumed by any external skill directly. Defined at `tools/check-prelude-triggers/src/triggers/mod.rs:16`.

## LoadError

TOML-loader error enum for the tool's argv-supplied input files (context-map, pipeline-stages, financial-precision-crates, changed-paths). Two variants via `thiserror::Error`: `Io { path, source: std::io::Error }` for filesystem-access failures and `Toml { path, source: Box<toml::de::Error> }` for parse failures. Propagated to `main.rs` where it terminates the process with a non-zero exit and a structured error line on stderr. Defined at `tools/check-prelude-triggers/src/toml_io.rs:11`.

**Duplication note.** This `LoadError` is the `check-prelude-triggers` TOML loader error type. A lexically identical `LoadError` exists in `cfdb-concepts` (spec'd at [`specs/concepts/cfdb-concepts.md`](../concepts/cfdb-concepts.md)) with its own variants for concept-overrides + published-language-crates loading. The duplication is intentional per the tool's source-file rationale at `tools/check-prelude-triggers/src/toml_io.rs:3-4` — `tools/check-prelude-triggers/` does NOT depend on `cfdb-concepts` by design, and the two types share a pattern shape (`thiserror` over `Io` + `Toml`) but not a definition or a dependency. Each bounded context owns its own error vocabulary; `audit-split-brain` reports the duplicate-type-def and marks it as a documented clone.
