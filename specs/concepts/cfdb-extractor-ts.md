# Spec: cfdb-extractor-ts

TypeScript `LanguageProducer` MVP (RFC-041 Phase 3 / issue #265). Walks a tsconfig.json-style TS project via tree-sitter-typescript and emits the v0.1 cfdb fact set onto the existing closed-set schema vocabulary in `cfdb-core::schema::labels` — no new `:Item.kind` values are introduced (Published Language invariant per RFC-041 §4). Operationally a cfdb-monolith member crate per `.cfdb/concepts/cfdb.toml`; gated behind `lang-typescript` in `cfdb-cli` so default builds drop the entire tree-sitter dep tree.

## TypeScriptProducer

Reference implementation of `cfdb_lang::LanguageProducer` for TypeScript workspaces. `name()` returns `"typescript"` (matching the `lang-typescript` Cargo feature on `cfdb-cli`). `detect()` requires BOTH `tsconfig.json` AND `package.json` at workspace root — the tsconfig signals "this is a TS project" and the package.json gives module-resolution context, distinguishing TS from a plain JavaScript project that just happens to ship `package.json`. `produce()` walks `**/*.ts` under workspace_root (skipping `node_modules/`, `dist/`, `build/`; `.tsx` deferred to a follow-up alongside JSX-component support), parses each file via tree-sitter-typescript's `LANGUAGE_TYPESCRIPT` constant (0.23 ABI shared with `cfdb-extractor-php`), and emits sorted-canonical `(Vec<Node>, Vec<Edge>)`. TS → Rust closed-set mapping (preserves the Published Language invariant; documented in the crate-root doc): TS `module` (file-level) → `:Module` node; TS `interface` → `:Item { kind: "trait" }` (contract w/o impl); TS `type alias` → `:Item { kind: "type" }` (exact semantic match); TS `class` → `:Item { kind: "struct" }` (stateful aggregate); TS `exported function` → `:Item { kind: "fn" }`; TS `const` / `let` / `var` → `:Item { kind: "const" }`.
