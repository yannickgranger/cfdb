# Spec: cfdb-lang

The producer-side seam for cfdb's multi-language story — defines the `LanguageProducer` trait that every language-specific producer (Rust today via `cfdb-extractor`, PHP and TypeScript follow via #264 / #265) implements, plus the `LanguageError` enum that flows through it.

`cfdb-lang` is the **language production** sub-concern of the cfdb monolith — a logical seam in the producer-side architecture, distinct from `cfdb-core`'s schema-vocabulary concern (which owns the `:Item` / `:CallSite` / `:Module` / `:Crate` types + the `IN_*` / `INVOKES_AT` edge labels) and from `cfdb-extractor`'s Rust-producer concern (which owns the syn walker + the `Cargo.toml`-based detector). Operationally it belongs to the same `cfdb` bounded context as every other cfdb-* crate per `.cfdb/concepts/cfdb.toml` (council-cfdb-wiring §B.1.2 — the eight cfdb crates form a single bounded context for ownership / concept-registry purposes); the "language production" framing is the architectural sub-context, not the operational one. Both layers matter: the architectural separation is load-bearing per RFC-041 §3.2 (the trait is a producer-side port, not a schema-vocabulary owner — concrete producers MUST emit only the closed-set `:Item.kind` values declared in `cfdb-core::schema::labels`), while the shared operational context keeps `BELONGS_TO -> :Context` resolution coherent across the cfdb monolith (`cfdb-extractor`'s self-workspace assertion at `crates/cfdb-extractor/tests/self_workspace.rs:244` requires every cfdb-* crate to resolve to `context:cfdb`). New `:Item.kind` values for future languages (`"interface"` for TypeScript, `"trait_php"` for PHP traits) require a separate schema RFC + `cfdb-core::SchemaVersion` patch + lockstep PR on `graph-specs-rust` per RFC-033 §4 I2 — Published Language invariant per RFC-041 §4.

Phase 1 stability profile (per RFC-041 §3.2 SAP table): Ca = 2 (`cfdb-extractor` + `cfdb-cli` depend on the trait), Ce = 1 (`cfdb-core` for `Node` / `Edge` types), I = 0.33, A = 0.50 (1 trait + 1 enum, both abstract), D = 0.17 — Zone of Usefulness. Adding more producers (#264 PHP, #265 TypeScript) decreases instability further.

## LanguageProducer

The producer-side port. Object-safe trait with `Send` supertrait bound (no `Sync` — the v0.1 dispatcher is single-threaded sequential per RFC-041 §3.1; re-add only when polyglot-parallel-dispatch is designed). Three methods: `name(&self) -> &'static str` returns the kebab-case identifier matching the `lang-<name>` Cargo feature on `cfdb-cli`; `detect(&self, &Path) -> bool` is a cheap marker-file check that gates the expensive walker; `produce(&self, &Path) -> Result<(Vec<Node>, Vec<Edge>), LanguageError>` walks the workspace and emits the v0.1 fact set. The method name is `produce` (not `extract`) per ddd-specialist R1 — escapes the four-way homonym across the CLI verb (`cfdb extract`), the crate name (`cfdb-extractor`), and the legacy fn (`cfdb_extractor::extract_workspace`). Object safety satisfied under the standard conditions: no generic methods, no `where Self: Sized` clauses, all method receivers are `&self`, no associated types — verified by the `language_producer_is_object_safe` test pin in `crates/cfdb-lang/src/lib.rs:tests`. `Box<dyn LanguageProducer>` is the dispatch shape used by `cfdb-cli`'s `available_producers()` registry. Defined at `crates/cfdb-lang/src/lib.rs:62`.

## LanguageError

The error type a `LanguageProducer` may surface — three variants. `NotDetected { producer: &'static str, reason: String }` fires when `produce()` is invoked on a workspace the producer's `detect()` would have rejected (callers who bypass the dispatcher: tests, embedding). `Io(#[from] std::io::Error)` covers I/O failure walking the workspace. `Parse { producer: &'static str, message: String }` covers producer-specific parse failure — for the Rust producer this wraps `cfdb_extractor::ExtractError`'s rendering; for future producers it wraps their parser's message. The `producer:` field is `&'static str` because every concrete producer's `name()` returns a static string slice — propagating it through the error preserves the diagnostic chain without forcing heap allocation on the error path. The string-shape `message` (rather than boxing the original error type) is the right shape for an object-safe trait: consumers cannot generically introspect concrete error types through `&dyn LanguageProducer`, so a stable string carrier is the minimum-faithful representation. Defined at `crates/cfdb-lang/src/lib.rs:113`.

## RustProducer

The reference implementation of `LanguageProducer` for Rust workspaces — owned by `cfdb-extractor` (RFC-041 §3.3 / Slice 41-B). `name()` returns `"rust"` (matching the `lang-rust` Cargo feature on `cfdb-cli`). `detect()` is the marker-file check `workspace_root.join("Cargo.toml").is_file()`. `produce()` delegates to the legacy `cfdb_extractor::extract_workspace` and wraps any `ExtractError` into `LanguageError::Parse { producer: "rust", message: e.to_string() }`. The legacy public function `extract_workspace` is preserved verbatim (same signature, same `ExtractError` return) for backward-compat with qbot-core's `cargo install --git` consumer per RFC-041 §3.3 + EPIC #279. Its deprecation timeline is a deferred architectural decision (RFC-041 §6); revisit when qbot-core moves off git-pinned source OR when EPIC #279 lands a migration shim. Defined at `crates/cfdb-extractor/src/lib.rs:99`.
